Redis + Caching
1 ms
or 1 s?

Your database is not slow.
Your hot path is expensive.

For repeated reads, caching turns a trip to durable storage into a memory lookup.

01 / 10
The definition

Redis is a fast, in-memory data store often used as a shared cache.

A cache keeps a disposable copy of data close to the code that needs it.

Fast because

Most reads and writes happen in RAM, not on a database disk path.

Safe to lose because

The database remains the source of truth for cacheable application data.

Mechanics

A key maps to a value, usually with an expiry.

1 Application asks for user:42
2 Redis checks its in-memory keyspace
3 A TTL removes stale entries automatically
4 Redis may evict keys when memory is full

Common Redis structures: strings, hashes, lists, sets, sorted sets, streams.

05 / 10
Applied example

A product page asks for the same catalog record thousands of times.

Without cache

Every visitor consumes a database connection and query for mostly unchanged data.

With cache

Redis absorbs repeated reads; the database handles misses and writes.

Cache stable, read-heavy data.
Skip caching what must be exact right now.

Good candidates: product details, rendered fragments, rate-limit counters, session data, leaderboard rankings.

06 / 10
Pseudocode

Keep cache logic at the read boundary.

key = "product:" + product_id cached = redis.get(key) if cached: return decode(cached) product = database.find(product_id) redis.set(key, encode(product), ex=300) return product

This is cache-aside: application code owns the fallback and population behavior.

07 / 10
Trade-off

Caching buys speed by accepting that copies can be old.

Gain

  • Lower latency
  • Fewer database reads
  • Higher burst capacity

Give up

  • Perfect freshness
  • Simple invalidation
  • Unbounded memory

Choose a TTL intentionally

It is a product decision: how stale may this data be before a user notices or is harmed?

08 / 10
Common failure

“We added Redis, so the database will be fine.”

×

Cache stampede

One popular key expires. A crowd of requests misses together and all hit the database at once.

Reduce the blast

Add TTL jitter so hot keys do not expire in lockstep.

For the hottest keys

Coalesce refreshes or serve a slightly stale value while one request refills it.

Also avoid caching errors indefinitely and remember to invalidate after meaningful writes.

09 / 10
Interview follow-up

“How would you keep cached data fresh after an update?”

Concise answer

Write the database first, then delete or update the corresponding cache key. A later read repopulates it. Use a TTL as a backstop, not as the only correctness mechanism.

Cache invalidation is not magic. It is a consistency contract.

State the acceptable staleness, choose an invalidation path, and measure hit rate, latency, evictions, and memory.
