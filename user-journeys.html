Apollo

One topic. Ten slides. A deck you can open anywhere.

Apollo is a Codex-native workflow that turns a single software-engineering topic into a self-contained, interview-ready slide deck. No external model, no API, no runtime — just a topic in Codex and a validated deck.html with ten PNGs out.

How Apollo Delivers

1. Start with one topic

Ready
  • Invoke Apollo ($apollo) with a single software-engineering topic.
  • Broad topics get an interview-relevant angle; narrow topics get deeper treatment.
  • No project runtime exists yet — this runs inside a Codex session.
  • Every run has a caller-supplied unique run-id; its deck.html and ten PNGs live in runs/<run-id>/ (no shared or cwd output folder).

2. Author in Codex

Ready
  • Codex's available model authors the content and visual HTML directly — Apollo has no separate model or API.
  • Each slide follows the fixed pedagogical order as an internal content-planning constraint — no separate outline artifact exists.
  • A single checked-in frame template locks the header, footer, visual feel, type, and colors and declares a body-safe area; the author composes the body freely within it.

3. Produce a self-contained deck

Ready
  • deck.html has no external assets, no network calls, no interactivity or animation.
  • Each slide is 1080×1350 CSS pixels.
  • The supplied reference HTML informs tone and visual guidance; its external assets are not copied.

4. Review and revise (advisory)

Growing
  • A checked-in manifest sets independent content and visual revision limits (each 0–5).
  • Content review checks a correct explanation, a concrete example, a trade-off or failure mode, and interview-ready Q/A; visual review reads the rendered PNGs for frame integrity, legibility, and collisions.
  • Reviewers report feedback to the author, who revises the deck HTML. On exhaustion the run still delivers with reports under runs/<run-id>/reviews/; review is advisory, not a hard gate.

5. Validate the deck

Ready
  • Exactly 10 top-level slides; each 1080×1350; no overflow; no external dependencies.
  • Invalid or incomplete decks are rejected — Apollo fails clean rather than ship them.
  • The validity contract is the shape of the deliverable, not a feature to toggle.

6. Export ten PNGs

Ready
  • Export exactly ten 1080×1350 PNGs, predictably named runs/<run-id>/slide-01.png through runs/<run-id>/slide-10.png.
  • The structural validator runs first (scripts/check-deck.py), then a local Node Playwright export script rasterizes slides 1–10 and validates count and 1080×1350 dimensions.
  • Network is disabled and the viewport is 1080×1350 at device scale 1; the run fails clean with no partial slide PNGs if either is wrong.

7. Keep docs and terminology aligned

Ready
  • $context keeps docs/ARCHITECTURE.md, docs/CONTEXT.md, and this map in sync after settled decisions.
  • docs/PRODUCT.md holds product intent; implementation stays in specs.
  • Prefer surgical doc updates over sprawling rewrites.
