Apollo Architecture Reset — E2E First

2026-07-19 · Architecture Review Report

Apollo is over-seamed for one fixed theme. First recover a truthful, runnable render contract, then remove complexity. This report proposes no code changes — only a prioritized roadmap and ticketed candidates.

Key finding — documentation-vs-runtime drift, not confirmed breakage: The composer TOML defines the agent’s write boundary as slide-bodies/. The render SKILL describes the same agent outputting composition.html. The runtime scripts expect composition.html to exist with [data-slide] elements. Specs 0004/0005 describe slide-bodies/ fragments assembled deterministically into index.html. No E2E failure has been recorded or proven. The drift means the SKILL prose is stale relative to the TOML agent boundary, but the runtime may still function if the assembly step (not the agent) produces composition.html. P0.0 must confirm this with a real smoke run.

Current vs Target Pipeline

Current (as documented across sources)

request
  → writer / retries
  → validate
  → reviewer / revisions
  → art director / layout
  → composer (writes slide-bodies/ per TOML; SKILL says composition.html)
  → composition.html (runtime assembly artifact)
  → recovery / snapshots
  → index.html (byte-identical copy) / export / manifest

Target

request
  → writer
  → content validator
  → optional human review
  → composer writes fragments
  → deterministic assembly → index.html
  → preflight
  → PNG export + manifest

The target pipeline has no recovery stage, no auto-revision loops, no art-director/layout step, and no durable composition.html. All intermediates are private; index.html is the sole published HTML artifact. Deterministic gates stop and return diagnostics when content is malformed.

  • ✗ Reject — No orchestrator, plugin/theme framework, new dependency, or schema migration.
